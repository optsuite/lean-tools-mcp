## Phase 0：`$/lean/tryTactics` endpoint

### 动机

原来的 `lean_multi_attempt` 工具需要逐个修改文件内容来尝试不同 tactic，是串行的 O(N) 操作。新增 `$/lean/tryTactics` endpoint 后，所有 tactic 在服务器内部并行执行，延迟降为 O(1)。

### Lean 源码改动（3 个文件）

**`src/Lean/Data/Lsp/Extra.lean`** (+14 行)
- 新增 `TryTacticsParams` 结构体：继承 `TextDocumentPositionParams`，增加 `tactics : Array String`
- 新增 `TacticResult` 结构体：包含 `tactic`、`goals`（成功时的剩余目标）、`error`（失败时的错误信息）

**`src/Lean/Server/FileSource.lean`** (+3 行)
- 新增 `FileSource TryTacticsParams` 实例，使 Watchdog 能正确路由请求到对应 FileWorker

**`src/Lean/Server/FileWorker/RequestHandling.lean`** (+46 行)
- `import Lean.Elab.Tactic.Basic`
- 新增 `handleTryTactics` 函数：
  1. 通过 `findGoalsAt?` 复用现有基础设施获取 cursor 处的 goal 状态
  2. 提取 `ContextInfo`（含不可变 Environment）和 `TacticInfo`（含 `mctxBefore`、`goalsBefore`）
  3. 为每个 tactic 启动 `IO.asTask`，在共享 `ContextInfo` 上独立运行
  4. 每个 task 内部用 `try-catch` 捕获错误，互不影响
- 在 `builtin_initialize` 中注册 `$/lean/tryTactics` handler

### 核心原理

```
Client → $/lean/tryTactics { position, tactics: ["simp","ring","omega"] }
  │
  ├─ findGoalsAt? → GoalsAtResult { ctxInfo, tacticInfo }
  │
  ├─ IO.asTask: run "simp"  on shared ctxInfo (独立 MetaM 状态)
  ├─ IO.asTask: run "ring"  on shared ctxInfo (独立 MetaM 状态)
  └─ IO.asTask: run "omega" on shared ctxInfo (独立 MetaM 状态)
  │
  └─ 收集所有结果 → Array TacticResult
```

- **共享**：`ContextInfo`（含 `Environment`、`LocalContext`）和 `mctxBefore` —— 不可变值，零成本共享
- **隔离**：每个 `IO.asTask` 有独立的 `MetaM` 状态（可变 `MetavarContext` 副本）
- **并行度**：N 个 tactics = N 个 Lean Tasks，由 Lean runtime 调度到 OS 线程

## Phase 1：FileWorker try-catch 容错

### 动机

原来 FileWorker 的 `mainLoop` 如果遇到未捕获异常，整个 worker 进程会 crash，然后 Watchdog 重启进程并重新加载 imports。对 Mathlib 项目来说，重启一次 ~30-60 秒。

### Lean 源码改动（1 个文件）

**`src/Lean/Server/FileWorker.lean`** (+8 行改动, -5 行)

#### 改动 1：`mainLoop` 为每个 handler 加 try-catch

| 消息类型 | 原来 | 现在 |
|---------|------|------|
| `Message.request` | 直接调用 `handleRequest`，异常冒泡 crash | `try handleRequest catch → 返回 LSP InternalError 响应 + 日志` |
| `Message.notification` | 直接调用 `handleNotification`，异常冒泡 crash | `try handleNotification catch → 日志记录` |
| 无效消息 | `throwServerError` 直接 crash | 日志记录 + 继续循环 |

#### 改动 2：`initAndRunWorker` 外层 catch 不再 re-throw

- 原来：`throw err` → 杀掉进程 → Watchdog 重启 → 重新加载 Mathlib (~30s+)
- 现在：记录日志 + 发送 diagnostic → 进程优雅退出
- 通常不会走到这里，因为 `mainLoop` 内部已经 catch 了

### 效果对比

| 场景 | 原来 | 现在 |
|------|------|------|
| 请求 handler 抛异常 | worker crash → 重启 (~30-60s) | catch → 返回错误响应 → 继续 (~0ms) |
| notification handler 失败 | worker crash → 重启 | catch → 日志 → 继续 |
| 无效 JSON-RPC 消息 | `throwServerError` crash | 日志 → 继续 |
| `mainLoop` 本身异常 | `throw err` → 进程退出码 1 | 日志 → 优雅退出码 0 |

核心收益基于 Lean Environment 的 persistent data structure 特性：snapshot 中的 `Environment` 是不可变的。elaboration 只创建新的 `Environment`（扩展旧的），不修改旧的。所以 "恢复" 就是丢弃失败状态——零成本。
