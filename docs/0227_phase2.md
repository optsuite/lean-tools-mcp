# Phase 2 修改记录：FileWorker 从进程改为 Task + Environment 共享

## 概要

Phase 2 将 Lean 4 Server 的 FileWorker 从独立 OS 进程改为进程内 Task，共享 imported Environment。
这是内存优化的核心改动，使得打开 N 个文件时只需加载 1 份 Environment，而非 N 份。

## 架构变化

```
改前 (N 个进程):
[Client] ↔ [Watchdog 进程]
                ├── Process.spawn → [FileWorker A] (import Env ~347 MB)
                ├── Process.spawn → [FileWorker B] (import Env ~347 MB)
                └── Process.spawn → [FileWorker C] (import Env ~347 MB)
             内存: Watchdog + N × Env

改后 (1 个进程, N 个 Task):
[Client] ↔ [Watchdog 进程]
                ├── IO.asTask → [Worker Task A] ──┐
                ├── IO.asTask → [Worker Task B] ──┤── 共享 Environment
                └── IO.asTask → [Worker Task C] ──┘
             内存: Watchdog + 1 × Env + N × 小量
```

## 内存测试结果

| 指标 | 进程模式 | In-process 模式 | 节省 |
|------|---------|----------------|------|
| 5 文件总内存 | 2018.9 MB | 592.7 MB | **70.6%** |
| 第 1 个文件增长 | +437.8 MB | +391.0 MB | — |
| 第 2-5 个文件增长 | ~348 MB/file | ~2.8 MB/file | **99.2%** |
| 进程数 | 6 (1+5) | 1 | — |

第 2 个及之后的文件几乎零增长（~2-3 MB），说明 Environment 缓存和共享工作正常。

## 启用方式

```bash
# 启用 in-process worker 模式
LEAN_WORKER_INPROCESS=1 lean --server

# 默认仍使用进程模式（完全向后兼容）
lean --server
```

## 修改的文件

### Phase 2a: Channel ↔ Stream 适配器
**新文件: `src/Lean/Server/ChannelStream.lean`**
- `channelInputStream`: 将 `Std.Channel String` 包装为可读 `FS.Stream`（含 buffer 管理）
- `channelOutputStream`: 将 `Std.Channel String` 包装为可写 `FS.Stream`（含 flush 机制）
- `sendLspMessage`: 辅助函数，序列化 LSP 消息并通过 Channel 发送
- EOF 哨兵检测：收到空字符串时抛出 EOF 异常

### Phase 2b: FileWorker 结构体重构
**修改: `src/Lean/Server/Watchdog.lean`**
- 新增 `WorkerHandle` 归纳类型 (`.process` | `.inProcess`)
- `FileWorker` 存储 `handle: WorkerHandle` + 预绑定的 `stdinStream`/`stdoutStream`
- `waitForProc`/`killProcAndWait`/`tryKill` 按 handle 类型分派

### Phase 2c: In-process FileWorker 启动
**修改: `src/Lean/Server/Watchdog.lean`**
- `ServerContext` 新增 `leanOpts: Options` 字段
- 新增 `startFileWorkerInProcess`: 通过 Channel + `IO.asTask` 启动 worker

**修改: `src/Lean/Shell.lean`**
- 传递 `opts.leanOpts` 给 `watchdogMain`

### Phase 2d: 模式切换与消息路由
**修改: `src/Lean/Server/Watchdog.lean`**
- `ServerContext` 新增 `useInProcessWorkers: Bool`（读 `LEAN_WORKER_INPROCESS` 环境变量）
- `startFileWorker` 重构为调度器 → `startFileWorkerProcess` / `startFileWorkerInProcess`
- 提取 `finishFileWorkerSetup` 共享后续初始化逻辑
- EOF 哨兵机制：worker task 在 `finally` 中发送空字符串

### Phase 2e: Environment 共享

- `importsLoadedRef` 从全局改为 per-worker 参数

### Phase 2f: 边界情况修复

## 已知限制

1. **IO.setStderr 竞争**：多个 in-process worker 会覆盖全局 stderr 前缀，仅影响日志格式，不影响功能
2. **Task 无法强制终止**：in-process worker 不支持 SIGKILL 式强制终止，依赖 EOF 哨兵和异常传播
3. **单进程风险**：如果 worker 中的 native code 死循环或栈溢出，会影响整个 Watchdog 进程（进程模式下只影响单个 worker）
